01. Introduction

When was the last time you printed a webpage?

It was probably to print a recipe or set of instructions. Maybe it was a google doc, complete with url detritus along the footer. Even people who have been using the internet since the era of tables and inline styles can probably count the number of times they’ve found it useful on one hand. Web layouts simply aren’t meant to be navigated in a fixed, sequential space and trying to print one results in a garbled, embarrassed version of its former self.

html & css are the primary tools for building a website; one is content, the other is styling. In desktop publishing terms, html is hammering keys to write an article and css is applying paragraph styles and kerning headlines. These languages can be pointed at a print layout just as easily as the screen by nesting css styles inside the selector @media print, a tool so often overlooked. If this seems like meaningless jargon but you’re interested in the relationship between web and print, hold tight.

The @media selector is normally used for detecting mobile devices and delivering a suitable layout for the viewer, but specifying that the media is print (rather than screen) can create a website that is as agreeable live-coded as hard-copied. This book aims to demonstrate a simple print stylesheet anyone can use to create texts, manuals, articles, comics, art books—anything the user might like to distribute.

When searching for new information, the general reflex is to find it on the internet. It’s available anywhere and has a massive, instant return compared to libraries or asking other people. While the internet is not always the best resource, it’s undeniably the first for most cases. A printable website means that a reader can find it through google, be linked to it by a friend, or stumble upon it in its original form, and still walk away with a hard copy that has all the material benefits of print. It’s available offline, it can be passed to someone over coffee, it’s easily referenced and flipped-through. A print-out would always has its online companion should a web format suddenly become better suited.

From the very start of developing a print framework, binding was an important aspect. In an age of infinite information, printed matter we bring into our homes is reserved for special objects. Why buy a paperback you’ll struggle to find shelf-space for when you can download it onto an e-reader for a fourth of the price? (Actually, there are a lot of answers to that question, but you’ll be spared a software monopoly tirade for the moment.) Throwaway content like news and opinion pieces, disposable in its relevance more than its quality, simply does not need to take up paper and ink anymore. When is print no longer the cheapest way to deliver textual content, it is freed of the obligation to do so—and so we have enjoyed magazines with experimental layouts, rich paper stocks, and niche interests. Sure, you could print a website with optimized or even beautiful styling, but why keep around a stack of pages stapled at the corner like a syllabus? The idea of a website manifesting as a bound book with cover & spine is ineluctable. I imagine the thrill of an industrial designer with access to a 3D printer for the first time is similar.

The final (or at least full) version of this print framework will include binding instructions that can be hooked into the content as a chapter or section, should the author choose. There are a huge number of binding methods that can be done without any special materials, all with different aesthetic and practical qualities. A book of intaglio star chart reproductions may recommend the heaviest stock your printer can manage with a paper-lock case, whereas a textbook of Java programs that draw different cats may shrug and recommend printing in greyscale to save ink, staple if you please. Everything from saddle-stitch to a hardback Smythe-sewn is possible if a reader is determined to make a book appear where none existed previously, except in data.

02. Web vs. Print

Screens and Paper Have Different Protocols.

web vs. print

Barrier for Entry

Publishing on the web is different from print in almost every way except for the presence of a human creator and the written word. Even that isn’t guaranteed with the advent of embedded media and generative programming. Internet publishing has no printing costs, no chain of suppliers or distributors, and everything can be served from a remote web host with your domain slapped on the front. That’s not to say publishing online isn’t serious business, but the barrier for entry starts at about $9 a month these days (or free if you use a platform like Tumblr or Wordpress). Anyone can try their hand at being an independent publisher, and reach an incredibly wide audience without wrestling with international shipping.

This has created an interesting rift where older players have a print background they must translate to web, whereas their newer colleagues know web as the default and may be lost if they’re composing spreads in a magazine. Print and screen have similarities but their visual language and protocols are necessarily different. Most of us take these nuances for granted—hypothetically you will only notice poor publication design in either medium, whereas great design is glossed over comfortably. Have you ever had to zoom in on a mobile device because the text was so small and hard to read? Or been annoyed by a huge sans-serif font in a magazine? The designer was probably trying to tackle both mediums at once (a request that seems trivial until attempted), or struggled in the translation between standards.

Typography

It’s old wisdom that the most important aspect of typography to get right is your body text (text type) because it makes up 90% of the text on the page, even online. Elements like headlines, subtitles, and pull-quotes usually get to be treated with fancier or more interesting fonts (display type) but they’re rarely longer than a line or two, and readability is less stringent.

Serif typefaces were the first ever created, and there is a rich library to choose from for printed text type. Letters with serifs are easier to read because their letter-shapes are very distinct from each other: we can recognize the shape of an entire word quickly without confusion between I and l or p and q. Reading long blocks of text is very comfortable with a good serif face, and it’s hard to go wrong when choosing between classics like Caslon and Garamond which have been iterated upon for centuries.

Sans-serif typefaces are the staple for text type on screens. Most screens have a resolution of 72 dpi (dots per inch) and the fine details of serif typefaces become fuzzy, as pictured left in an exaggerated example. Hairlines and ball terminals look spotty and jagged, which is not only a disservice to the typeface but irritating to read. Sans-serifs, on the other hand, are usually fairly mono-weight and render much more faithfully on screens. There’s a reason the system font on your computer is Tahoma or San Francisco (maybe even Ubuntu); all sans-serifs designed for screens specifically.

A common design choice is for display type to be the opposite of text type in terms of serifs, and so you’ll often see a flip-flop between print and screen: screen text is sans-serif and print text is serif, and both will have the opposite for their display type. However, using serif typefaces has become more and more acceptable as screen resolutions improve, and web publishers are able to choose their font pairings with more options. The luxe, textural appearance of serif text type is finally an option for web publishing, and so the print-classic pairing of a display sans and text serif has made its way onto our screens. High resolution displays and ever-improving web fonts are allowing web publishers to experiment with print design vocabulary and blur the lines between what you might see scrolling online and what you might see turning the page in a magazine.

To make an example, this book is using Freight Text for text type and Apercu Pro for display. Freight Text was designed primarily for screen use, and comes packaged with the still-rare ability to render intermediate typography like small caps, swashes, and contextual alternates through css. Apercu Pro, on the other hand, is a beautiful display typeface and a horrible choice for body text. Its design would not be nearly as popular if we were still stuck using only sans-serifs just like it online. Being able to choose a serif text type like Freight Text also means accessing more expressive italics, as most sans-serifs have a bland italic font that is simply skewed to the right. It may even be called "oblique" instead of italic in the font files. There are exceptions, but the fact that italics are meant to be a different (albeit related) font entirely from their regular/Roman version is most apparent in serif typefaces.

Technical Differences

So, the world of typography has opened up for web publishers. Great! There are still limitations (web fonts can slow down loading times and don’t have as many bells & whistles as their desktop-to-print counterparts) but a huge gap has been bridged between web and print. However, it’s worth examining fundamental differences between the mediums so we can understand how they can be translated. This book has attempted to convert formal elements where an equilibrium (such as using serif text type) isn’t possible. While print has changed over time with innovations like desktop publishing and offset printing, web publishing is improving exponentially faster, as computer-based technology tends to do. Speed of innovation isn’t necessarily better; web development is constantly suffering growing pains as new standards and software are adopted slower than they are created. However, if something you want to create online isn’t possible, it’s never a long wait.

Links

Hyperlinks mean external sources can have their location embedded in the text, and being able to click on one and arrive somewhere new is a hallmark of web publication (and the internet in general). Print has several ways of referencing external sources so the purpose of the link has to be considered. An easy option is adding the actual url inline with a css pseudo class. If the html for a clickable link online looks like this: <‍a href=“http://google.com”>google<‍/a> adding a pseudo class in the @media print stylesheet which expands the print would look like this: a::after {
  content: " (" attr(href) ")";
} The content is whatever is inside of the href attribute in the last example. And now a printed link to google (http://google.com) expands itself.

The problem with this method is the assumption that your reader will put down the book, open a browser on a nearby device, and type the link in. It’s better than no link at all if there’s any ambiguity about what you’re referencing or how to find it, but it’s kind of like saying “here’s the source, I’ve done my journalistic duty, you can go find it if you really want”. It acts more like a Further Reading suggestion than a connected piece of information. Chicago-style footnotes and annotated citations, more commonly found in scholarly journals than magazines, do a much better job of providing context in the moment. In the print stylesheet, you can add a similar pseudo class that appends a footnote indicator instead of an entire link: a::after {
  content: “1”;
  font-size: 6pt;
  vertical-align: super;
} which adds a small numeral (in this case, 1) as superscript after your link. The footer of your page can now host not just the location of that source but a brief summaryor annotation. Attaching a custom class like “printOnly” and hiding it in the screen stylesheet means it will only reveal itself when the reader accesses the print stylesheet. An unordered list with two list items in the footer serves quite well: <‍footer>
  <‍ul>
    <‍li>1. Dr. Suess’s study concluded that the green-ness of breakfast food had no effect on its taste compared to un-dyed foods when subjects were blindfolded.<‍/li>
    <‍li>2. While it makes some ground-breaking discoveries regarding the social structures of Whos, the reader should be cautioned that Horton Hears a Who does use some racist and out-dated language.<‍/li>
  <‍/ul>
<‍/footer> which can then by styled in the screen stylesheet as footer {
  display: none;
} and in the print stylesheet as per your requirements—probably fixed to the bottom of the page with a rule divider and smaller text, but the world is your oyster.

The elegance of this solution is thwarted by the fact that any attempt to fix a footer to the bottom of the page does not work when printing without some front-end programming. As far as your browser is concerned, the bottom of the page is the bottom of the entire web page, not where paper ends or page breaks are inserted into the code. Mozilla and Chrome make use of the css selector @page, which allows styling page-specific values like margins and size, but its footer and layout functionality is limited.

The more advanced your solution is, the more likely it is you’ll have to employ programming languages like Javascript or Ruby, which allow the use of much more complex logic and parameters to construct the page. At risk of scaring anyone away from learning these very useful languages, adding scripts to your page increases its complexity on more than one level. However, it does mean you can easily insert content into “templates” you’ve written yourself. Web programming generally lets you be more prescriptive and yet more flexible in how content is retrieved and arranged.

Book Anatomy

Traditional print media—specifically books and magazines—has an anatomy of parts that isn’t often mirrored online. Because it exists as a sequential structure without links for direct navigation, things like a table of contents and an index are necessary. This is where the creative coding of a website becomes more valuable. With some class selectors and page breaks you can turn a horizontal navigation bar into a vertical table of contents and entire sections can be swapped with more print-appropriate content. A Further Reading section might link to web articles online and books that can be found in a local library in print. Showing and hiding elements that only work in one medium (video, for example) can become an embedded game of hide and seek.

The most disappointing pitfall of using @media print is the lack of flexibility in styling elements as ink on a page. Regular print can take advantage of sprawling spreads and predictable text layout, but using a print stylesheet hooked up to html means everything is rendered one page at a time, with no reference to things on pages that came before or after. Developing a print stylesheet for this book made it apparent that the existence of things like @page and recto-verso margins are luxuries in the first place. However, this does put some creative constraints on developers and authors which may forge something more interesting than having free reign in inDesign.

Making this book

Printing this page works best in Google Chrome or Firefox because of the use of @page selectors, which is not supported in Safari or Internet Explorer. The latter two can still be used but the layout will not be optimized for recto-verso pages.

You can always print this book as a stack of pages to be stapled together. That’s the most common application of printing a webpage and it should suit you just fine.

The following will cover how to get the pages out of your computer and prepared for binding. In the interest of multimedia, and because using a chocolate box as a book cover is adorable, a concise video tutorial by Sea Lemon will walk through how to bind those pages with glue and cover.

Perfect binding with sections

Perfect binding is a method of bookbinding you’ve probably interacted with a lot. It’s used for every magazine with a thick, flat spine and most soft-cover books. It’s also incredibly simple once you have your pages printed and folded, and can be used to bind all sorts of paper configurations if you need something held together in a pinch without staples.

You will need

  • Printer that can print double-sided
    or “duplex”
  • PVA glue
    the cheap white stuff you used in elementary school
  • Bulldog clips, a big hair clip, or a chip-bag clip
    or a book press if you’re already book-making inclined
  • Paintbrush
    one you’re not too attached to

Navigate to File > Print... in your browser’s menu.

Click "Print using System Dialogue..." and in this new window, tick two-sided.

The system dialogue is your computer’s method of printing documents, instead of the browser’s, which usually means more custom options. A lot of control has been relinquished by your browser now, and we’re limited to what the printer and your OS know how to do.

In the Layout sub-menu, select Long-Edge binding. In some printers this can actually bind it, but we’re just indicating how the back-side of a page should be oriented.

Only print pages 3 to the end, as pages 1 and 2 are for the cover. The first page should be the title page.

If you’re printing smaller than letter-size, select a thin hairline border.

This acts as crop marks in lieu of actual printer’s marks, so you know where to trim the pages on letter-size paper. This book is formatted for its ideal size to be A5.

If you don’t have a duplex printer, print even pages first and odd pages second to imitate the effect.

To “manually” print two-sided pages, print even-numbered pages in reverse order, put the pages back in the paper feed, and then print odd-numbered page in normal order.

If you’re using an inkjet printer that finishes printing face-up and has a paper feed in the rear, print the even-numbered pages in normal order, re-feed, and then print the odd-numbered pages in reverse order.

Print the cover on heavy paper like cardstock or watercolour rag.

Because the cover has to wrap around the pages entirely, you'll need to print on a tabloid sized page and trim. Print 2 pages per sheet, and print pages 1 to 2 with reversed order—this should render in the preview as the cover graphic on the right and a blank "page" on the left.

With two-sided printing, this book is about 16 sheets of paper, which means you'll need to trim the width of the cover to be 2-times the page width plus 2-3 mm for the spine, although the best way to get the spine size right is to fold it around your specific stack of papers being bound.

Follow Sea Lemon’s instructions for perfect binding.

This is where you’ll need the bulldog clips, PVA glue, and brush. From there it takes about 15 minutes minus drying time.

03. making your own book

code online & print offline.

Making Your Own Book

With more context, it can be reiterated that the goal of this publication is to serve as a first draft for a web-to-print development framework. Several online publications have created print stylesheets but they more-or-less just strip the web version of anything that won’t print well, such as links and navigation bars. CSS-Tricks1, a site filled with tutorials and tips for working with css, has one that aims to strip away print-unfriendly content and make the text as readable as possible. They know their pages will be printed as offline reference material, and creating a beautiful print layout would require more wrangling than their readers really need.

Many web publications do put a lot of sweat and tears into providing both formats in their optimized glory, but they are developed separately—the print version is a product for purchase put together by print designers. e-flux journal has a (rather genius) pdf generator2 with a variety of templates but it’s proprietary software.

A web framework can have many forms but the simplest is a pre-made css stylesheet. Bootstrap3 and Foundation4 both have dense stylesheets you can reference in your html code, and styling a page is as simple as slotting classes into tags that match your purpose, such as “navbar” or “button”. And, of course, it’s always possible to add your own css on top as a separate stylesheet should you want to customize it further.

As it stands, this books’s stylesheets are far too customized for public use (though feel free to peak at it via right click > View Page Source; they’re linked inside the <‍head> tag). Experimenting with all the abilities and limitations of @page and @media print meant some jerry-rigging and the use of magic numbers. A final public version will be released by the end of 2015, available on GitHub5. However, html and css are quick to get started with if you’re not yet familiar with them, and the ability to create and modify your own content online is incredibly liberating.

Of Interest to the Reader f

Triple Canopy

Triple Canopy is a web-first magazine that puts out print compendiums of its online content as a means of archival rather than as a primary format. Its articles scroll horizontally, with columns and images re-flowed to fit your viewport.

Flaneur

Flaneur, named for the French concept of the modern, meandering “man of the streets”, examines one city-street each issue with a literary focus. It has a section of web-specific content called the Unprintables, taking advantage of the multi-media abilities of our computers.

Uncube

Uncube is a Berlin-based architectural magazine that puts out a monthly issue much like print magazines, but keeps them entirely online. Articles can be browsed by thumbnail, an attractive feature of hyperlinks.

The Gourmand

The Gourmand, describing itself as a food & culture journal, asks readers to choose between pring and screen right from the homepage. The content is entirely different between the two, one taking advantage of full print spreads and the other image slideshows and videos.

The Binder and the Server

The Binder and the Server is an essay by senior editor of Triple Canopy Colby Chamberlain, published in Art Journal’s Winter 2011 volume. It is the result of discussion among members of Triple Canopy about how and why they tackle the medium of web publication, and what has changed since a generation raised with the internet took to their places in the publishing world.

Sea Lemon’s Book-binding Tutorials

Sea Lemon, a DIY youtube channel, has a fantastic playlist of book-binding tutorials. The videos are bright, straight-forward, and include alternatives for book-binding tools that you may not have on hand or want to invest in for just one book.

04. Further Reading

Of Interest to the Reader f

Protocol: How control exists after decentralization

Alexander R. Galloway, 2004

Protocol explains how strict structures and systems must exist for networks to function, and that the internet is built on control, not freedom. Computers are incredibly literal, which means those of us who interface with them must obey protocols if we hope to communicate effectively, such as how a browser renders code or what software is installed on a reader’s computer. This very book follows strict css protocols so that something unusual and creative—printing it—is possible (although not without its own frustrations).

Print is Dead. Long Live Print: The world’s best independent magazines

Ruth Jamieson, 2015

Ruth Jamieson has collected examples of independent magazines succeeding where Big Print fails in a series of case studies titled Print is Dead. Long Live Print. Jamieson believes that print as we knew it for hundreds of years has been slain by web publishing, but like a forest fire its death clears the way for new growth. Today’s magazines can afford to be unconcerned with immediacy, universal appeal, or competing with the internet’s torrent of information. Niche, material, and innovation are king.

Making Handmade Books: 100+ bindings, structures & forms

Alisa Golden, 2010

Making Handmade Books is filled with concise and illustrated instructions for (literally) hundreds of binding methods and variations. There’s always youtube, but being able to browse so many methods in one place without sifting through the shit of the internet is incredibly useful if you’d like to try your hand at more than one.

Publication Design: A guide to page layout, typography, format and style

Allen Hurlburt, 1971

Publication design is such a wide and varied topic that it’s hard to begin recommending books about it. However, Hurlburt’s book is classic and, being written before the advent of web publication, can provide some insight into how print worked back then when it wasn’t influenced by or competing with screens(besides some very early-stages desktop publication).

05. glossary

web & print designers use a lot of jargon.

glossary of terms

  • HTML

    stands for Hyper Text Markup Language. html is the base language websites are written in. Content is wrapped in “tags” which indicate to the browser what sort of element it is, such as a header, footer, link, etc.

  • CSS

    stands for Cascading Style Sheet. css is a coding language that selects elements in a website’s html and dictates what that element should look like and how it affects other elements. Giving the header a white background with text inside it aligned to the right would look like: header {
      background-color: #ffffff;
      text-align: right;
    }     html is the content, and CSS is how it looks.

  • Perfect Binding

    method of binding a book by aligning loose sheets of paper along their intended spine and applying glue, usually with a brush or palette knife. Thick magazines and soft-cover books are usually bound this way. A cover and even a dust-jacket can be added.

  • Section

    the “section” of paper you fold together like a taco to create facing pages. Usually this is for smythe-sewn binding but sometimes perfect binding too. Sections are stacked and sewn together or otherwise bound through their fold edges. Some forms of Japanese binding face the section fold towards the fore-edge of the book to keep pages double-layered, usually because of the use of extremely thin paper.

  • Fore-edge

    the edge of a page that faces away from the spine. This is the edge you usually grab to turn the page.

  • Head

    the top edge of a page.

  • Tail

    the bottom edge of a page.

  • Media Query

    a CSS statement that asks a browser what device or media the browser is serving your content to, such as screen or print. All the styles wrapped inside the query will apply if the media matches its rules; a media query of @media screen and max-width < 480px will have its styles applied to the page if the device is a screen and less than 480 pixels wide, such as an iPhone 5 or most Android devices.

  • Front-end Pro-
    gramming

    html & css are not programming languages, but rather mark-up and styling languages. The word “programming”, like “font”, is used incorrectly a lot because of the mythos around "coding" and computer science in general. Programming languages like Javascript or Ruby can use variables, complex logic, and algorithms to serve content dynamically and contextually. Front-end programming refers to programming for web design specifically; the front-end is what the user sees and interacts with.

  • Internet Browser

    While we use browsers all the time, from a development perspective you need to know that they don't just connect us to the internet, they render a website's code as layouts and visuals and interactive elements. Each browser is made by a different company or organization, and while they try to be consistent with each other, they all have different philosophies and standards about how things should be rendered. This is the source of Internet Explorer being a groan-worthy joke; it renders code so differently from other browsers that developers often ignore its requirements entirely or put in lower-quality alternatives for IE users to save a headache.

06. bibliography

citations, references, & annotations.

  1. https://css-tricks.com/.
    CSS-Tricks is a blog dedicated to CSS and ways it can replace programming like Javascript and PHP.

  2. http://supercommunity.e-flux.com/issue/.
    e-flux journal distributes the print version of their publication by sending PDFs to publishers made from their PDF generator. The generator itself was written by Adam Florin in Ruby—at the time of printing this book, his blog post about its development is offline but was previously located at http://adamflorin.com/work/e-flux-journal-layout-generator.

  3. http://getbootstrap.com/.
    Bootstrap was developed by Twitter as a column-based web framework and is one of the most popular ever. It exists as a very long stylesheet containing hundreds of styling options as classes which can be attached to your html. You’ve probably used hundreds of websites built with builtstrap, generally characterized by its "jumbotron" landing image with a carousel cycle of images.

  4. http://foundation.zurb.com/.
    Foundation is a web framework like Bootstrap, with less focus on styling typography and other aesthetic elements for you. It is also incredibly popular—the popularity of both an indicator of what poor-shape column-based layouts on the web are in. While they may seem the default they can require some taming and jerry-rigging.

  5. https://github.com/laurelobrien.
    That’s me! Github is a version-control and collaboration tool. With Git you can “commit” changes to a file or project you can roll back or compare changes between versions, and Github is the online tool that syncs it all for public browsing. Github is one of the cornerstones of open-source software today; everything you want to sync with Github is public, and the entire interface is based around downloading, cloning, pushing changes to other people’s projects and accepting feedback. The entire project folder for this book is viewable there, as will be the final public-use version of the print stylesheet.
